# Horizontal vs. Vertical Scaling

## 1. WHY
Every system starts small. But what happens when your user base grows 10x overnight? Your system needs to handle more load — and how you handle that growth is one of the most fundamental decisions in system design.

This topic comes up in almost every system design interview because every large-scale system has had to make this decision. Getting this wrong means downtime, data loss, or massive unnecessary cost.

## 2. WHAT
Scaling means increasing a system's capacity to handle more load.

### Vertical Scaling (Scale Up)
- Make the existing machine more powerful.
- More CPU, more RAM, faster disk — same machine, just bigger.

### Horizontal Scaling (Scale Out)
- Add more machines to share the load.
- Same size machines — just more of them working together.

Simple analogy:

You run a restaurant. Orders are piling up.
- Vertical → Replace your one chef with a superhuman chef who works 10x faster.
- Horizontal → Hire 10 regular chefs and split the orders between them.

## 3. HOW
### Vertical Scaling — How it works:
Before:  [Server — 8 CPU, 16GB RAM]
After:   [Server — 32 CPU, 128GB RAM]

- You upgrade the hardware
- Application code doesn't change
- No distribution logic needed
- But there's a hard ceiling — machines only get so big
- And it's still a Single Point of Failure

### Horizontal Scaling — How it works:
Before:
Client → [Server 1]

After:
Client → [Load Balancer] → [Server 1]
                         → [Server 2]
                         → [Server 3]

- A Load Balancer sits in front and distributes incoming requests
- Each server handles a portion of the traffic
- If one server dies → others keep running
- You can keep adding servers as load grows

## 4. DIAGRAM
### VERTICAL SCALING
─────────────────────────────────
  [Small Server]  →  [BIG Server]
   8CPU/16GB          64CPU/512GB
      ↑
   Same machine, upgraded
   Still ONE point of failure

### HORIZONTAL SCALING
─────────────────────────────────
              [Load Balancer]
             /       |        \
      [Server1]  [Server2]  [Server3]
       8CPU        8CPU        8CPU

   → Add Server4, Server5 as needed
   → No single point of failure

## 5. REAL-WORLD USAGE
| Company | How they scale |
| --- | --- |
| Netflix | Thousands of small servers horizontally scaled on AWS. Adds capacity during peak hours (evenings). |
| Twitter/X | Horizontal scaling with sharded databases to handle tweet volume |
| Early-stage startups | Often start with vertical scaling — simpler, cheaper initially |
| MySQL (databases) | Traditionally scaled vertically — harder to distribute |
| Cassandra / DynamoDB | Built from ground up for horizontal scaling |

## 6. TRADE-OFFS
| Factor | Vertical | Horizontal |
| --- | --- | --- |
| Complexity | Low — no code changes | High — need load balancers, distributed logic |
| Cost | Expensive hardware | Cheaper commodity machines |
| Limit | Hard ceiling exists | Virtually unlimited |
| Fault Tolerance | Single point of failure | Resilient — one node dies, others serve |
| Speed to implement | Fast | Slower — more infrastructure |
| State management | Easy — one machine | Hard — sessions/state must be shared |

### When NOT to use Horizontal Scaling:
- Early-stage product with low traffic — over-engineering kills speed
- Stateful systems that are hard to distribute (e.g., some legacy databases)
- When your team lacks infrastructure experience — operational complexity is real

### When NOT to use Vertical Scaling:
- When you're already near hardware limits
- When you need fault tolerance
- When traffic is unpredictable and spikes frequently

## 7. INTERVIEW FOCUS
Interviewers expect you to:

- Never just say "add more servers" — explain the load balancer, statelessness, and trade-offs
- Know that databases are harder to scale horizontally than app servers
- Understand the concept of stateless servers — horizontal scaling only works cleanly when servers don't store session data locally
- Mention auto-scaling — cloud platforms (AWS, GCP) can automatically add/remove servers based on load
- Know that most real systems use both — vertical up to a point, then horizontal beyond that

## 8. QUIZ
Answer all of these before we move on:

### Q1: Why is horizontal scaling harder for databases than for application servers?
A1: Databases are harder to scale horizontally because they manage state. When you distribute database workload across multiple machines, you have to decide where data lives, keep replicas consistent, and handle concurrent writes and reads across nodes. This creates data consistency challenges and requires solutions like sharding, replication, and careful coordination.

### Q2: Your app servers are horizontally scaled across 5 machines. A user logs in on Server 1 but their next request goes to Server 3, which has no session data. What is this problem called and how do you solve it?
A2: This is called session affinity or sticky sessions. Better solutions are:
- Use a centralized session store such as Redis so all servers share the same session data, or
- Use stateless authentication like JWT tokens so the request contains its own identity.

Note: Sticky sessions are a weak solution because they can cause uneven load distribution and still fail if the assigned server goes down.

### Q3: (Scenario) You're designing a system that expects steady low traffic for 11 months but a massive spike every December (like a tax filing system). Which scaling strategy would you recommend and why?
A3: I would recommend horizontal scaling with auto-scaling. It allows the system to stay small and cost-efficient most of the year, then automatically add capacity when demand spikes. Vertical scaling is limited by hardware ceilings and cannot shrink back easily after the peak.

### Q4: A teammate says "Let's just vertically scale our database — it's simpler." You're expecting 50x growth in the next year. How do you respond?
A4: Vertical scaling may be simpler in the short term, but at 50x growth it is risky. It creates a single point of failure and hits hardware limits quickly. A stronger long-term plan is to prepare for horizontal scaling with read replicas for reads and sharding for write scale, while architecting the system for distributed operation.

### Q5: What does it mean for a server to be stateless, and why does it matter for horizontal scaling?
A5: A stateless server does not keep any user-specific session data locally between requests. Each request must contain all the information needed to process it. This matters because horizontal scaling only works well when any server can handle any request, regardless of which server handled the previous request.

## Additional points to remember
- The core issue with horizontal database scaling is data consistency and state management, not just that databases store data.
- A centralized session store or JWT is a stronger solution than sticky sessions.
- Auto-scaling is the best argument for horizontal scaling in bursty workloads.
- Mentioning read replicas and sharding shows understanding of database scaling beyond a single server.
- Stateless servers are essential because they allow flexible request routing and resilience when servers are added or removed.
