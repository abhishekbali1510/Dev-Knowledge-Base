# What is Capacity Estimation?

## 1. WHY
Before you design any system, you need to know how big it needs to be. Without this, you're guessing — and guessing leads to either:

- Over-provisioning → Wasting millions on infrastructure you don't need
- Under-provisioning → System crashes under real load, users leave

In interviews, capacity estimation shows the interviewer that you think before you build. It signals seniority. Most junior engineers skip this and jump straight to drawing boxes. Senior engineers size the system first.

It also directly influences every decision after this — how many servers, what database, whether you need caching, etc.

## 2. WHAT
Capacity estimation is the process of calculating the resources your system needs based on expected usage.

Resources means:

- Traffic — How many requests per second?
- Storage — How much data will you store?
- Memory — How much RAM for caching?
- Bandwidth — How much data flows in and out per second?

You're not expected to be exact. You're expected to be reasonably accurate within an order of magnitude — and show structured thinking.

## 3. HOW
There are 4 numbers you estimate in almost every system:

### Step 1 — Daily Active Users (DAU)
Start with users. Everything else derives from this.

Example: Design Twitter
- Monthly Active Users = 500 million
- DAU = ~20% of MAU = 100 million users/day

### Step 2 — Requests Per Second (RPS)
Assume each user performs ~10 actions/day (tweets, likes, reads).

- Total requests/day = 100M users × 10 = 1 billion requests/day
- RPS = 1,000,000,000 / 86,400 seconds
- ≈ 11,500 RPS
- Peak RPS = ~2x to 3x average = ~25,000 RPS

Always calculate peak RPS, not just average. Systems crash at peak, not average load.

### Step 3 — Storage Estimation
Example: Each tweet = 280 characters ≈ 300 bytes
- metadata (user ID, timestamp) ≈ 200 bytes
- Total per tweet ≈ 500 bytes

- Tweets per day = 100M users × 1% post = 1M tweets/day
- Storage/day = 1M × 500 bytes = 500 MB/day
- Storage/year = 500 MB × 365 ≈ 180 GB/year
- Storage/10 years ≈ 1.8 TB (text only)

If media (images/video) is involved, multiply by 100x easily.

### Step 4 — Bandwidth Estimation
Outgoing (read) bandwidth:
- Each user reads ~100 tweets/day
- Each tweet = 500 bytes

- Bandwidth = 100M users × 100 tweets × 500 bytes
- = 5,000,000,000,000 bytes/day
- = 5 TB/day
- = ~58 MB/second outgoing

## 5. REAL-WORLD USAGE
| Company | Estimation Challenge |
| --- | --- |
| WhatsApp | 100B messages/day — storage and bandwidth at extreme scale |
| YouTube | 500 hours of video uploaded every minute — storage estimation is critical |
| Uber | Millions of location pings per second from drivers — RPS dominates |
| Instagram | Photos are ~3MB each — storage grows faster than user count |

These companies run capacity planning teams whose only job is this — at scale, getting it wrong costs millions of dollars per day.

## 6. TRADE-OFFS
| Aspect | Reality |
| --- | --- |
| Precision | You will never be exact — and that's fine. Order of magnitude is the goal |
| Assumptions | Always state them clearly. Interviewers reward visible reasoning |
| Read vs Write ratio | Most systems are read-heavy (Twitter: 100:1 read/write). This changes your design |
| Growth factor | Always ask — is this day 1 or year 3? Design for future load, not current |
| Media vs text | Media storage grows exponentially faster — always separate these estimates |

## 7. INTERVIEW FOCUS
Interviewers expect you to:

- Ask clarifying questions first — DAU, read/write ratio, data size
- Never skip this step — jumping to design without estimation is a red flag
- Show your math out loud — even rough numbers with clear reasoning score well
- Use round numbers — 100M not 97.3M. Speed matters in interviews
- Connect estimates to design decisions — "Given 25K RPS, we'll need a load balancer and at least 10 app servers"
- Know key constants by heart:
  - 1 day = 86,400 seconds ≈ 100K seconds (rounded)
  - 1 month = ~30 days
  - 1 year = ~365 days
  - 1 char = 1 byte
  - 1 image = ~300KB to 3MB
  - 1 video (min) = ~50MB to 500MB
  - 1 server = ~1K to 5K RPS (commodity)

## 8. QUIZ
Answer all of these:

### Q1: Why do we calculate peak RPS instead of average RPS when deciding server count?
A1: We calculate peak RPS because systems fail at peak load, not average load. Designing for peak gives you a safe capacity buffer so servers can handle sudden bursts without crashing.

### Q2: A system has 50 million DAU. Each user sends 2 messages per day. Each message is 1KB. Calculate:
- Requests per second
- Storage per day
- Storage per year

A2:
- Total requests/day = 50M × 2 = 100M requests/day
- RPS = 100M / 86,400 ≈ 1,157 RPS ≈ ~1,200 RPS
- Peak RPS = ~2,500 to 3,600 RPS
- Storage/day = 100M × 1 KB = 100 GB/day
- Storage/year = 100 GB × 365 = 36,500 GB/year = 36.5 TB/year

### Q3: (Scenario) You're designing a photo sharing app like Instagram. A teammate says storage won't be a problem because "text is cheap." What's wrong with this reasoning?
A3: The reasoning is wrong because photo storage is far larger than text storage. Media files grow much faster than users, so storage estimates must include images and videos separately. Text is cheap, but media is costly and dominates the total storage footprint.

### Q4: Your system gets 10x more reads than writes. How should this ratio influence your architecture decisions?
A4: A 10:1 read/write ratio means the system should optimize for read scale. Use read replicas for the database, add a large cache layer, and move frequently requested content out of the primary DB. This reduces load on the write path and improves read performance.

### Q5: Why is it important to state your assumptions out loud during capacity estimation in an interview?
A5: Stating assumptions out loud shows the interviewer your reasoning and prevents misunderstandings. If your assumptions are wrong, the interviewer can correct them early, and you avoid building the wrong design on top of bad numbers.

## Additional points to include
- Always divide by 86,400 seconds when converting daily requests to RPS.
- Convert units to readable terms like GB or TB in interviews.
- For media-heavy systems, separate text and media storage estimates to avoid underestimation.
- Read-heavy systems need caching and read replicas, not just bandwidth planning.
- Interviewers want to see your assumptions, not just your final numbers.
