# What is HTTP & HTTPS?

## 1. WHY
Every system you design has clients talking to servers. Understanding how that communication works at the protocol level is fundamental. HTTP is the backbone of the web — every API call, every webpage load, every mobile app request uses it.

In interviews, this comes up when discussing:

- API design
- Security of data in transit
- Why certain systems use WebSockets or gRPC instead
- How load balancers and CDNs work

If you can't explain how clients and servers talk, you can't design the communication layer of any system.

## 2. WHAT
### HTTP (HyperText Transfer Protocol)
A set of rules that defines how a client and server communicate over the internet. It is stateless — every request is independent, and the server remembers nothing about previous requests.

### HTTPS (HTTP Secure)
HTTP + TLS (Transport Layer Security) encryption on top. The data transmitted is encrypted so no one in the middle can read it.

Think of it like this:

- HTTP = Sending a postcard. Anyone who handles it can read it.
- HTTPS = Sending a letter in a sealed, tamper-proof envelope. Only the recipient can open it.

## 3. HOW
### HTTP Request-Response Cycle
1. Client opens a TCP connection to the server
2. Client sends an HTTP Request
3. Server processes it
4. Server sends back an HTTP Response
5. Connection closes or is kept alive for reuse

### Anatomy of an HTTP Request
- Method → GET, POST, PUT, DELETE, PATCH
- URL → /api/users/123
- Headers → Content-Type, Authorization, etc.
- Body → (for POST/PUT/PATCH) the actual data being sent

### Anatomy of an HTTP Response
- Status Code → 200 OK, 404 Not Found, 500 Server Error
- Headers → Content-Type, Cache-Control, etc.
- Body → The actual response data (JSON, HTML, etc.)

### Common HTTP Methods
| Method | Purpose | Example |
| --- | --- | --- |
| GET | Fetch data | Get user profile |
| POST | Create new data | Submit an order |
| PUT | Replace existing data | Update entire profile |
| PATCH | Partially update data | Change just the email |
| DELETE | Remove data | Delete a post |

### Common Status Codes — Know these cold
| Code | Meaning |
| --- | --- |
| 200 | OK — success |
| 201 | Created — new resource created |
| 301 | Moved Permanently — redirect |
| 400 | Bad Request — client sent wrong data |
| 401 | Unauthorized — not logged in |
| 403 | Forbidden — logged in but no permission |
| 404 | Not Found |
| 429 | Too Many Requests — rate limited |
| 500 | Internal Server Error |
| 503 | Service Unavailable |

### How HTTPS Works — TLS Handshake
1. Client says "Hello" and sends supported encryption methods.
2. Server responds with its SSL certificate and public key.
3. Client verifies the certificate against trusted Certificate Authorities.
4. Client and server agree on a shared secret key.
5. All further communication uses symmetric encryption with that shared key.

Simplified flow:

Client                          Server
  |                               |
  |--- "Hello, I support AES" -->|
  |                               |
  |<-- "Here is my certificate" --|
  |                               |
  |--- "Here is a secret encrypted with your public key" -->|
  |                               |
  |<======= Encrypted Data ======>|

## 5. REAL-WORLD USAGE
| Company/System | How they use it |
| --- | --- |
| Banks/payment systems | HTTPS mandatory — credit card data in transit must be encrypted |
| Google | Switched entirely to HTTPS in 2014; now ranks HTTPS sites higher in search |
| Netflix | HTTPS for all API calls between client and server |
| Load Balancers (AWS ALB) | SSL termination — HTTPS handled at the load balancer, internal traffic can be HTTP |
| CDNs (Cloudflare) | Terminate TLS at edge and serve cached content over HTTPS |

SSL Termination is an important concept:

Client ──HTTPS──► [Load Balancer] ──HTTP──► [Internal Servers]
                  (decrypts here)

Internal network is trusted, so re-encrypting every internal hop may be unnecessary overhead.

## 6. TRADE-OFFS
| Factor | HTTP | HTTPS |
| --- | --- | --- |
| Security | None — plaintext | Encrypted in transit |
| Performance | Slightly faster | TLS handshake adds latency (one-time cost) |
| Cost | Free | SSL certificates (mostly free now via Let's Encrypt) |
| SEO | Penalized by Google | Preferred |
| Trust | No padlock | Browser shows 🔒 |

When might you use HTTP internally?
- Service-to-service communication inside a private VPC/network where traffic never leaves your infrastructure
- Avoids TLS overhead on high-frequency internal calls
- Still debated — zero-trust security models argue even internal traffic should be encrypted

## 7. INTERVIEW FOCUS
Interviewers expect you to:

- Know the difference between HTTP and HTTPS clearly
- Understand TLS handshake at a high level — no cryptography depth needed
- Know HTTP methods and when to use each
- Know status codes — especially 401 vs 403, PUT vs PATCH
- Understand statelessness of HTTP and why that matters for scaling
- Know SSL termination at load balancers — comes up in almost every design
- Understand HTTP/1.1 vs HTTP/2 at a basic level:
  - HTTP/1.1 → One request per TCP connection (slower)
  - HTTP/2 → Multiple requests over the same connection (multiplexing — faster)
  - HTTP/3 → Built on UDP instead of TCP (even faster, used by Google)

## 8. QUIZ
### Q1: What is the difference between 401 and 403 status codes? Give a real-world scenario for each.
A1: 401 means authentication is missing — the user needs to log in or provide credentials. Example: Accessing /dashboard without a valid token.

403 means the user is authenticated but does not have permission. Example: A regular user trying to access /admin.

### Q2: Your system makes 10,000 internal service-to-service calls per second inside a private network. A security engineer says all internal traffic must use HTTPS. A performance engineer says it's unnecessary overhead. Who is right and why?
A2: Both are right depending on context. Security-wise, HTTPS or mTLS is safer and follows zero-trust principles. Performance-wise, internal traffic can be faster without TLS. In a sensitive environment, use TLS internally. In a trusted private network where performance is critical and the infrastructure is secured, SSL termination with HTTP internally is a reasonable trade-off.

### Q3: What does stateless mean in the context of HTTP, and what problem does it create for systems that need to remember who a user is?
A3: Stateless means the server does not store request-specific data between requests. Each request is independent. The problem is the server cannot remember that a user already logged in, so the client must send authentication data each time, such as a JWT token or session identifier stored in a shared session store.

### Q4: (Scenario) A junior engineer on your team uses GET requests to delete a user account because "it works the same way." What is wrong with this and why does it matter?
A4: GET should be safe and not modify state. Using GET to delete data breaks HTTP semantics and can cause serious issues: browser prefetching, crawlers, and proxies may replay or cache the request, leading to accidental deletions. The correct method is DELETE (or POST if DELETE is not available), because it clearly signals a destructive operation.

### Q5: What is SSL termination and why do large systems do it at the load balancer instead of at each application server?
A5: SSL termination means the load balancer decrypts HTTPS traffic before forwarding it to internal servers. Large systems do this to centralize decryption, reduce CPU overhead on app servers, and simplify certificate management. App servers then receive plain HTTP requests from a trusted internal network, which is faster and easier to scale.

## Additional points to remember
- 401 = authentication missing; 403 = authenticated but forbidden.
- Mention zero-trust models when discussing internal encryption.
- Stateless HTTP is what enables clean horizontal scaling.
- GET is safe by definition and should never perform destructive actions.
- SSL termination at the load balancer is a common optimization, but end-to-end TLS is used when security demands it.
