# What is System Design?

## 1. WHY
Every software system you use — Instagram, Uber, WhatsApp — was designed before it was built. As systems grow to serve millions of users, decisions about architecture, data flow, and infrastructure become critical. A wrong design choice early on can cost months of rework or cause the entire system to collapse under load.

In interviews, system design tests your engineering judgment — not just whether you can code, but whether you can think at scale.

## 2. WHAT
System design is the process of defining the architecture, components, and data flow of a system to satisfy given requirements.

Think of it like being an architect designing a building. You don't lay bricks yourself — you decide:

- How many floors?
- Where are the exits?
- How does electricity and water flow?

In software, you decide:

- How do clients talk to servers?
- Where is data stored?
- How does the system handle 10 million users?

## 3. HOW
Every system design problem follows a basic thought process:

Requirements → Components → Data Flow → Scale → Trade-offs

Example — Design a URL Shortener (like bit.ly):

1. User pastes a long URL
2. System generates a short code
3. Stores the mapping (short → long) in a database
4. When someone visits the short URL → look up → redirect

Simple at 100 users. But at 1 billion requests/day, you now need to think about:

- Caching (avoid hitting DB every time)
- Database choice (SQL vs NoSQL)
- How to generate unique short codes fast

## 6. TRADE-OFFS
System design is never about a "perfect" solution — it's about conscious trade-offs.

| Choice | Gain | Lose |
| --- | --- | --- |
| More servers | Higher availability | Higher cost |
| Caching | Faster reads | Data can be stale |
| NoSQL DB | Flexible & scalable | Weaker consistency |

A senior engineer always asks: "What are we optimizing for?"

## 7. INTERVIEW FOCUS
Interviewers are NOT looking for the perfect answer. They want to see:

- Can you clarify requirements before jumping in?
- Can you identify bottlenecks proactively?
- Do you understand trade-offs of your choices?
- Can you evolve your design when constraints change?

The thought process matters more than the final diagram.

## 8. QUIZ
Take your time and answer these:

### Q1: What is the difference between functional and non-functional requirements? Give an example of each.
- Functional requirements describe what the system must do.
  - Example: "User can place an order."
- Non-functional requirements describe how well the system performs or behaves.
  - Example: "The system should handle 1 million requests/day."

### Q2: A system works perfectly for 1,000 users but crashes at 1 million. What category of problem is this, and what does it tell you about the original design?
- This is a scaling problem.
- It shows the original design was not built for horizontal scalability, load distribution, or higher traffic. It may rely too much on a single server, lack caching, or have a database that cannot handle the increased load.

### Q3: Your manager says "just use a single powerful server, it's simpler." What are the risks of this approach?
- The biggest risk is a single point of failure: if that server fails, the entire system fails.
- Vertical scaling has hard limits and cannot support infinite traffic.
- There is no redundancy, no fault isolation, and no way to absorb traffic spikes safely.

### Q4: You're designing a food delivery app like Zomato. Name 3 core components you'd need and what each one does.
- Order Management: stores orders and tracks status from placed to delivered.
- Delivery Agent Matcher: finds the nearest available delivery person and assigns orders.
- Real-Time Updates: pushes live status changes to users via notifications or WebSockets.
